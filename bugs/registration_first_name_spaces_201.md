## Название

Регистрация: `first_name` с пробелами в начале или в конце принимается с **201 Created** вместо **400 Bad Request**

## Шаги воспроизведения

1. Отправить `POST /api/registration/` с валидными `email` и `phone_number`.
2. В поле `first_name` указать значение с **пробелом в начале** строки (например, ` Иван`) или с **пробелом в конце** (например, `Иван `).
3. Убедиться, что ответ имеет статус **201** и тело успешной регистрации.

## Ожидаемый результат

- API возвращает **400 Bad Request** (ошибка валидации имени).
- Пользователь с «грязным» именем (ведущие/замыкающие пробелы) **не создаётся**, либо сервер **нормализует** значение (trim) и явно документирует это поведение.

## Фактический результат

- API принимает запрос и возвращает **201 Created**: регистрация проходит так, как если бы имя было валидным без лишних пробелов по краям.

## Окружение

- Среда: dev
- URL: https://api.dev.ads.ktsf.ru
- Метод: POST /api/registration/

## Приоритет

- **P1**: некорректный ввод проходит как успешная регистрация — риск «мусорных» данных и расхождений при поиске/отображении пользователей без явного контракта на trim.

## Серьёзность

- **High**: ошибка валидации бизнес-данных (имя) приводит к **успешному созданию сущности** вместо отклонения запроса; влияет на качество данных и согласованность с ожидаемыми правилами имени.
Примечание: 
- Ошибка воспроизводится частично: большинство невалидных значений имени отклоняются (400), однако значения с пробелами в начале или конце строки по-прежнему принимаются с 201.
